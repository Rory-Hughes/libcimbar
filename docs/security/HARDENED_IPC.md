# Hardened OIP-to-SCP IPC Contract

## Scope

This contract defines the narrow byte protocol between the optical ingress
process (OIP), the sandboxed decoder boundary, and the Secure Core process
(SCP). It carries corrected fountain frames into the decoder domain and exact
opaque completed objects back to the Secure Core. It does not carry filenames,
paths, application commands, wallet instructions, nested objects, raw pointers,
or free-form diagnostics.

The parser implementation is `cimbar::parse_hardened_ipc_message` in
`src/profile/hardened_transport/HardenedTransportIpc.cpp`. It is independent of
OpenCV, Reed-Solomon, Wirehair, filesystem, zstd, and the Linux sandbox probe.

## Envelope

Every IPC message is a single fixed header followed by an optional opaque byte
payload. All integer fields are little endian. Messages are rejected unless the
entire input is exactly one envelope.

| Offset | Size | Field | Rule |
|---:|---:|---|---|
| 0 | 4 | Magic | ASCII `LCIP` |
| 4 | 1 | Version | Exact value `1` |
| 5 | 1 | Message type | One fixed value from the table below |
| 6 | 2 | Flags | Must be zero |
| 8 | 8 | Transfer generation | Nonzero unsigned integer selected by SCP |
| 16 | 4 | Payload size | Exact remaining byte count |
| 20 | 4 | Status code | One fixed signed code from the table below |

The header size is 24 bytes. There is no variable-length header, extension
block, nested TLV, string field, pointer field, or diagnostic field.

## Message Types

| Value | Name | Direction | Payload | Status code |
|---:|---|---|---|---|
| 1 | `submit_frame` | OIP to decoder | Corrected fountain frame bytes | `ok` only |
| 2 | `cancel_transfer` | SCP to decoder | None | `ok` only |
| 3 | `reset_transfer` | SCP to decoder | None | `ok` only |
| 4 | `completed_object` | Decoder to SCP | Exact opaque completed object bytes | `ok` only |
| 5 | `transfer_status` | Decoder to SCP | None | Nonnegative fixed status |
| 6 | `transfer_failed` | Decoder to SCP | None | Negative fixed status |

Generation zero is reserved. SCP increments the generation whenever a new
transfer begins or a reset/cancel creates a new receiver state. Recipients drop
messages for an unexpected generation before acting on their payload.

## Status Codes

| Value | Name | Use |
|---:|---|---|
| 0 | `ok` | Successful command or payload-bearing message |
| 1 | `receiving` | Decoder is accepting frames for the generation |
| 2 | `completed` | Decoder has completed but object was not yet taken |
| 3 | `object_taken` | Completed object ownership already moved |
| 4 | `cancelled` | Generation was cancelled |
| -1 | `invalid_generation` | Message generation is stale or unexpected |
| -2 | `invalid_frame` | Frame envelope or decoder policy rejected input |
| -3 | `resource_limit` | Configured message, object, CPU, or memory limit hit |
| -4 | `sandbox_violation` | Child violated the sandbox contract or watchdog |
| -5 | `internal_error` | Non-diagnostic implementation failure |

Status and failure messages never include a text reason. Operators can map
these fixed codes to local logs outside the decoder domain, but the IPC payload
itself stays non-diagnostic.

## Size Limits

The parser accepts a caller-supplied `HardenedIpcLimits` object. The default
limits are:

| Limit | Default |
|---|---:|
| Maximum full IPC envelope | 64 KiB |
| Maximum `submit_frame` payload | 16 KiB |
| Maximum `completed_object` payload | 64 KiB |

The parser rejects inputs larger than the full envelope limit before routing.
It then applies message-specific payload limits. The product profile should set
the object limit to the Secure-Core-selected object class bound and should set
the frame limit from the selected fountain chunk size and maximum packets per
frame.

## Parser Invariants

The parser:

- Rejects null non-empty inputs, short headers, bad magic, unsupported version,
  reserved flags, unknown types, generation zero, unknown status codes, and
  length mismatches.
- Rejects payload bytes on control, status, and failure messages.
- Requires non-empty payloads on `submit_frame` and `completed_object`.
- Returns payload only as an offset and length into the caller-owned input.
- Allocates no memory and performs no filesystem, network, decode, or logging
  work.

`fuzz_hardened_ipc` exercises this parser independently under libFuzzer,
AddressSanitizer, and UndefinedBehaviorSanitizer. The deterministic corpus is
generated by `fuzz/corpus/generate.py` under `fuzz/corpus/hardened_ipc/`.
